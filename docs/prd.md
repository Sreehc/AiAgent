## 1. 产品概述

AiAgent V1 是一个面向技术管理员与业务分析用户的通用多智能体平台，首版以“深度研究报告”作为核心业务闭环，支持用户从提出问题开始，经过任务规划、检索增强、工具执行、结果汇总，最终得到结构化报告和图片产物。附件、工具输出和跨任务产物复用已进入统一产物体系。

本产品参考 `OWWZO/ai-agent` 的核心能力设计，但目标不是简单复刻代码，而是形成一个可部署、可扩展、可持续演进的产品化平台。V1 重点完成五条主链路：

- 登录与邀请注册
- 对话与任务调度
- 深度研究与报告生成
- RAG 知识库增强
- 图片生成与产物复用

V1 前端以登录态用户工作台为中心，优先保障真实使用链路闭环；账号体系采用账号密码登录、邀请制注册，默认区分管理员与普通用户两类角色，会话、知识库和产物默认归属个人用户。

## 2. 产品目标

- 构建一个支持多 Agent 协作的通用平台，而不是单轮问答机器人。
- 支持复杂任务的规划、执行、汇总与结果交付。
- 支持 RAG 工作区，让用户可上传知识库并将其绑定到会话任务。
- 支持 MCP 工具注册、发现与调用，扩展平台外部能力。
- 支持报告、图片、附件、工具输出与上下文片段沉淀，并支持跨任务复用。
- 支持执行过程持久化与历史回放，提升可解释性与可追踪性。
- 支持账号中心能力，包括个人资料、修改密码、找回密码、登录日志和退出登录。

## 3. 成功指标

- 用户可以完成“登录 -> 发起任务 -> 查看规划执行 -> 输出报告”的完整闭环。
- 用户可以上传知识文档并在后续研究任务中使用检索增强结果。
- 管理员可以完成模型配置、MCP 工具源配置和知识库管理的基础操作。
- 单个任务执行过程中的计划、工具调用、总结和最终产物可以被历史会话稳定回放。
- 图片生成工作区可以完成文本生图和参考图编辑；真实 provider 最终可用性取决于外部服务凭证和接口兼容性。
- 用户所有会话、知识库、产物默认只对本人可见，管理员可基于权限进行管理。

## 4. 用户角色

### 4.1 管理员

负责平台能力治理与系统配置，典型职责包括：

- 创建邀请并发放账号入口
- 配置大模型与嵌入模型
- 管理 MCP 工具源
- 管理知识库与文档
- 查看模型、MCP 和邀请码基础配置

### 4.2 普通用户

负责使用平台完成研究、分析、报告生成与图片生成，典型职责包括：

- 通过邀请注册并登录
- 发起研究任务
- 查看报告和可下载的对象存储产物
- 绑定知识库提升回答质量
- 浏览历史会话与打开已有产物
- 在账号中心维护个人信息和密码

### 7.6 关键领域类型

- `AgentRequest`：一次任务请求的统一输入。
- `AgentSession`：一次会话的上下文容器。
- `ExecutionPlan`：计划执行模式下的规划结果。
- `ToolInvocation`：一次工具调用的结构化记录。
- `ArtifactRef`：会话产物引用。
- `KnowledgeBase`：知识库实体及其绑定关系。
- `McpServerConfig`：MCP 工具源配置。
- `UserAccount`：用户账号与角色视图。
- `InviteToken`：邀请注册凭证。

## 8. 技术与架构考虑

### 8.1 技术栈

- 后端：Java 21、Spring Boot
- 前端：React
- 数据层：PostgreSQL 17.10 + pgvector 0.8.2
- 缓存：Redis 8.4.0
- 消息队列：Kafka 3.8.0
- 对象存储：MinIO RELEASE.2025-09-07T16-13-09Z
- 大模型接入：OpenAI 兼容模型接口

### 8.2 架构风格

后端采用 DDD 风格演进，默认分为：

- `trigger`：HTTP / SSE / Job 等协议入口
- `application`：Agent 调度、用例编排、流程组织
- `domain`：核心领域模型、运行时语义、执行账本、记忆、RAG
- `infrastructure`：DAO、外部服务、MCP、文件与存储适配
- `app`：启动与配置装配

### 8.3 前端形态

前端以登录态用户工作台优先，V1 至少包括：

- 登录与邀请注册入口
- 聊天工作台
- 知识库工作区
- 图片生成工作区
- 历史会话页
- 账号中心
- 基础配置页

### 8.4 部署导向

本 PRD 面向“可部署产品”而不是纯演示 Demo，因此实现阶段需考虑：

- 可独立部署的前后端结构
- 运行配置外置
- 关键执行链路的日志与问题定位能力
- 基础鉴权与密码安全措施

## 9. 非目标

- V1 不以通用聊天体验优化为目标，核心是多智能体研究工作台闭环。
- V1 不做多租户体系。
- V1 不做复杂 RBAC 权限模型。
- V1 不做企业级配置中心。
- V1 不做计费系统。
- V1 不做审批流。
- V1 不承诺移动端优先体验。
- V1 不在 PRD 中展开数据库表级设计。

## 11. 风险与开放问题

### 风险

- 多 Agent 执行链路较复杂，若缺少统一账本模型，后续历史回放会难以补齐。
- RAG、MCP、图片生成三类能力同时进入 V1，集成复杂度较高。
- 登录后使用能提升治理能力，但会提高首轮接入成本，需要邀请码发放流程配套。
- 若前端工作台与后台配置边界不清，可能导致 V1 范围继续膨胀。

### 开放问题

- V1 报告输出格式是否默认 HTML、Markdown 双格式，还是先落一种格式。
- V1 图片生成是否只接一个模型提供方，还是保留多模型切换能力。
- V1 RAG 的默认文档类型范围是否限制为 PDF、Markdown、TXT。
- V1 基础配置页是否仅面向管理员可见，还是对高级用户开放部分能力。

## 12. 最低交付闭环

AiAgent V1 的最低可交付标准不是页面数量齐全，而是以下闭环成立：

- 用户可受邀注册并登录。
- 用户可发起研究任务并看到执行过程。
- 系统可产出结构化报告或明确失败结果。
- 用户可在历史会话中回看执行过程与产物。
- 所有会话、知识库和产物默认按个人归属隔离。
