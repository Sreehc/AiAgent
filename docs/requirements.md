# AiAgent 前端 UI 升级需求

> 文档状态：基于 2026-06 前端代码审计、现有 PRD/UX 文档、UI 重构讨论和产品确认项整理。本文档聚焦“功能已基本完成后的前端视觉与体验升级”，不替代 `prd.md`、`ux-spec.md` 和 `ui-rebuild-recommendation.md`。

## 1. 背景

AiAgent 当前已经形成较完整的产品闭环：用户可登录、创建研究会话、运行 Agent 任务、绑定知识库、查看执行时间线、生成报告和图片、复用历史产物；管理员可配置模型、MCP 服务、邀请码、审计和 RAG 评估。

当前主要问题不在功能缺失，而在前端观感和体验一致性：

- 当前绿色/橙色主色偏传统运营后台，与 AI Agent / Research Workspace 的专业感不匹配。
- `tokens.css` 与 `theme.css` 存在两套 token 体系，CSS 与 Tailwind 组件之间容易产生视觉不一致。
- 页面大量使用“白底 + 边框 + 圆角 + Badge”的通用后台样式，导致研究工作台、知识库、图片工作室、管理页之间缺少层级差异。
- Workspace 的时间线、工具调用、证据、产物等核心能力已经存在，但呈现方式偏调试记录，尚未体现成熟 Agent 工作台的产品质感。
- 全量替换为 assistant-ui 不适合当前项目，因为现有项目是 Vite + React Router + Spring Boot 后端，且已有自定义会话、SSE、RAG、MCP、产物和审计链路；更适合保留业务系统，只在必要位置参考或局部引入成熟 Chat/Agent UI 模式。

## 2. 目标

- 将 AiAgent 从“功能完整的后台工具”升级为“专业、可信、好看的 AI 研究工作台”。
- 建立统一的视觉系统，包括色彩、字体、间距、圆角、阴影、状态、表格、表单、空态、加载态和错误态。
- 先产出高保真设计稿并完成评审，再进入前端代码实现。
- 在不破坏现有业务能力的前提下，重构核心页面的信息层级和交互体验。
- 优先提升研究工作台、知识库、图片工作室、历史回放和管理页的可读性、操作效率和完成度。
- 保留当前 React Router + Vite + Tailwind + Radix-style 组件体系，采用增量升级而非重写。

## 3. 用户

| 用户 | 需求 | 关注点 |
| --- | --- | --- |
| 普通研究用户 | 发起研究任务、查看执行过程、阅读报告、复用产物 | 信息清晰、状态可信、报告和证据容易理解 |
| 知识库维护用户 | 上传文档、查看索引状态、测试召回质量 | 文档状态明确、检索结果可比较、错误可恢复 |
| 图片生成用户 | 生成研究配图、查看历史、挂接会话产物 | 预览体验好、历史图片易浏览、结果元数据清楚 |
| 管理员 | 管理模型、MCP、邀请码、审计和 RAG 评估 | 风险突出、配置高效、状态和异常易定位 |
| 开发/维护者 | 在现有代码基础上持续扩展 UI | token 清晰、组件统一、页面改动可控 |

## 4. 使用场景

### 场景 1：用户发起一次研究任务

用户进入研究工作台，选择或新建会话，输入研究问题，选择执行模式和知识库，提交后查看 Agent 的执行过程、工具调用、证据、最终报告和产物。

期望体验：

- 用户能一眼看清当前会话、任务状态和主操作。
- 执行过程以 Agent feed / trace 的形式展示，而不是原始日志堆叠。
- 工具调用、证据、报告、产物各有清晰分层。

### 场景 2：用户管理知识库并验证检索

用户创建知识库，上传文档，查看索引状态，并用检索测试验证召回片段。

期望体验：

- 知识库列表、文档表格和检索测试形成完整 RAG cockpit。
- 文档解析/索引状态明确，失败原因可见。
- 检索命中以证据卡片展示，包含分数、来源、片段和引用信息。

### 场景 3：用户生成研究配图

用户进入图片工作室，输入 Prompt 或上传参考图，生成图片并查看历史结果。

期望体验：

- 图片预览区域成为页面视觉中心。
- 历史结果以高质量缩略图网格展示。
- 图片状态、尺寸、会话关联和失败原因清楚可见。

### 场景 4：管理员排查系统状态

管理员查看模型配置、MCP 服务、审计记录和 RAG 评估结果。

期望体验：

- 默认模型、异常 provider、MCP 健康状态、评估失败等高风险信息被突出。
- 审计和 RAG 评估不直接暴露原始 JSON，而是先展示结构化摘要，详情可展开。

## 5. 功能范围

### 5.1 全局视觉系统

- 重建色彩方案，采用 graphite/zinc 中性色 + restrained cyan/teal 强调色。
- 合并并规范 `tokens.css` 与 `theme.css` 的语义 token。
- 建立亮色与暗色主题的配对规则，默认主题跟随系统设置。
- 统一字体、字号、行高、数字样式和代码/ID 样式。
- 允许引入或托管新字体，例如 Geist、IBM Plex Sans 或 Satoshi；具体字体需在高保真设计阶段确认。
- 统一组件 radius、shadow、border、focus ring 和 active/pressed 反馈。

### 5.2 通用组件升级

- 优化 `Panel`、`Button`、`Badge`、`StatusPill`、`Table`、`Tabs`、`Alert`、`EmptyState`、`Skeleton`、`Dialog`、`DropdownMenu`、`CommandPalette` 等组件。
- Badge 默认从胶囊风格调整为更紧凑的状态标签。
- Button 和 IconButton 增加统一 hover、focus、active scale 反馈。
- Table 增强密度、对齐、数字样式和横向溢出处理。
- JSON 和调试信息默认折叠，以摘要优先。

### 5.3 页面体验升级

| 页面 | 范围 |
| --- | --- |
| 登录/注册/找回/重置 | 强化品牌第一印象，保留现有表单流程 |
| 研究工作台 | 重构为左侧会话、中间 Agent feed、右侧 Inspector Tabs 的信息架构 |
| 知识库 | 升级为 RAG cockpit，强化文档状态和检索证据 |
| 图片工作室 | 强化图片预览、生成状态、历史缩略图和元数据，并支持批量选择、收藏、对比视图和下载集合 |
| 历史回放 | 强化回放摘要、trace、产物复用入口 |
| 账号中心 | 将个人 API 配置、资料、安全和登录日志做清晰分组 |
| 模型配置 | 突出默认模型、测试状态和风险配置 |
| MCP 服务器 | 突出连接状态、工具发现、健康检查和工具测试 |
| 审计 | 从原始 JSON 列表升级为结构化表格 + 展开详情 |
| RAG 评估 | 增加指标卡、用例列表、评估历史和失败详情 |
| 404 | 提供更清晰的返回工作台和登录入口 |

### 5.4 assistant-ui 评估范围

- 不进行 assistant-ui 全量替换。
- assistant-ui 只作为交互参考，用于借鉴消息流、工具调用折叠、附件、运行状态等 UI 模式。
- 本阶段不做 assistant-ui 局部 PoC，不引入 assistant-ui runtime。

### 5.5 管理端仪表盘总览

- 管理页需要新增仪表盘式总览，用于聚合模型、MCP、审计、RAG 评估和系统风险状态。
- 总览页应优先展示默认模型、启用模型数、异常 provider、MCP 健康状态、近期失败任务、近期登录失败、RAG 评估失败和待处理风险。
- 总览页不替代现有配置页面，点击各指标应能跳转到对应管理模块。

### 5.6 审计与 RAG 评估结构化字段

审计记录优先结构化展示以下字段；接口暂不具备的字段可在详情展开中降级显示，不阻塞首版 UI：

| 审计类型 | 优先字段 |
| --- | --- |
| 用户审计 | 时间、用户、动作、目标对象、角色/权限变化、结果、IP |
| 任务审计 | 时间、用户、会话/Run、任务标题、执行模式、状态、耗时、错误摘要、工具调用数、产物数 |
| 工具审计 | 时间、服务、工具名、关联 Run、状态、耗时、错误类型、输入摘要、输出摘要 |
| 登录审计 | 时间、用户、IP、User Agent、结果、失败原因 |

RAG 评估优先结构化展示以下字段：

| 模块 | 优先字段 |
| --- | --- |
| 评估历史 | Eval ID、状态、Top K、知识库数量、用例数量、通过/失败数量、主要指标、创建时间 |
| 评估用例 | Query、启用状态、期望 citation、期望文本、最近结果、失败原因 |
| 检索命中 | Rank、Score、文件名、Citation ID、Section、Chunk、检索策略、内容预览 |
| 指标详情 | Hit Rate、Recall、Precision、MRR/NDCG、失败用例列表；若后端仅返回聚合 `metrics`，先解析可展示字段，原始 metrics 放入展开详情 |

## 6. 非目标

- 不重写后端业务能力。
- 不迁移到 Next.js App Router。
- 不使用 `npx assistant-ui init --yes` 改造当前项目。
- 不把当前系统降级为单一聊天应用。
- 不移除现有 RAG、MCP、图片生成、历史回放、账号中心和管理员能力。
- 不在本阶段引入大规模数据层重构，除非 UI 改造必须依赖。
- 不以营销落地页风格替代工作台体验。

## 7. 约束

- 当前前端技术栈为 React 19、React Router 7、Vite 6、TypeScript、Tailwind 3、Radix-style UI primitives。
- 当前后端为 Spring Boot，聊天和研究任务依赖自定义 REST/SSE 接口。
- 所有 UI 改造必须保留现有路由和核心业务流程。
- 需要兼容亮色和暗色主题。
- 默认主题跟随系统；用户手动切换后应持久化偏好。
- 需要保证桌面端优先体验，同时移动端不能出现内容重叠、横向滚动或关键操作不可达。
- 交互控件需要满足基本可访问性要求，包括键盘可达、focus visible、足够对比度和图标按钮 aria label。
- 修改应采用增量方式推进，避免一次性重写导致功能回归。
- UI 升级需要纳入端到端截图回归或 Playwright 检查。
- 每个阶段完成后至少通过前端构建，并对核心金路径进行手动验证。

## 8. 成功标准

| 类型 | 标准 |
| --- | --- |
| 视觉 | 全站主色、表面、边框、阴影、状态色统一；不再呈现明显的默认后台模板感 |
| 功能 | 登录、研究任务、SSE 流式执行、知识库、图片生成、历史回放、账号、管理员配置均无回归 |
| 工作台 | 用户能清楚区分任务输入、执行过程、工具调用、证据、报告和产物 |
| 知识库 | 用户能快速判断文档索引状态，并理解检索命中来源和质量 |
| 管理页 | 管理员能快速识别默认配置、异常状态、失败记录和风险项 |
| 可访问性 | 关键文本对比度达标，所有主要交互支持键盘操作和清晰 focus |
| 响应式 | 主要页面在桌面、平板和手机宽度下不出现文本溢出、重叠和不可操作控件 |
| 工程 | token 体系清晰，通用组件复用率提升，页面内重复样式和原始 JSON 展示减少 |
| 验证 | `frontend` 构建通过，核心页面完成浏览器走查，并通过 Playwright 截图回归或端到端检查 |

## 9. 待确认问题

以下问题已经确认：

| 问题 | 决策 |
| --- | --- |
| 新视觉方向 | graphite + cyan/teal |
| 默认主题 | 跟随系统 |
| 新字体 | 允许引入或托管 |
| 设计稿 | 先产出高保真设计稿，再进入代码实现 |
| assistant-ui | 只作为交互参考，不做局部 PoC |
| 研究工作台右侧 | 采用 Inspector Tabs |
| 管理页 | 新增仪表盘式总览 |
| 图片工作室 | 支持批量选择、收藏、对比视图和下载集合 |
| 审计与 RAG 字段 | 按本文档 5.6 的优先字段推进 |
| 回归验证 | 纳入 Playwright 截图回归或端到端检查 |

仍需在设计阶段细化：

- 最终字体选择和字重组合。
- 高保真设计稿覆盖范围，是先覆盖核心五页，还是一次覆盖全站。
- Playwright 截图回归的 viewport、主题和关键路径清单。
